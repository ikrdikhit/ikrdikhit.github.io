## 1. Font Basics on Linux

### Where Fonts Live

Linux organizes fonts across two main tiers. System-wide fonts — available to all users — are stored under `/usr/share/fonts/`. Fonts installed by a package manager (apt, pacman, dnf, etc.) land here automatically, typically organized into subdirectories by format or family, e.g., `/usr/share/fonts/truetype/` or `/usr/share/fonts/noto/`.

Per-user fonts belong in `~/.local/share/fonts/`. This is the modern XDG-compliant path and the one you should use. The older `~/.fonts/` directory still works on most distros, but it's legacy — new setups should default to `~/.local/share/fonts/`. You can create subdirectories inside either location freely; fontconfig scans recursively.

```bash
# Check what's installed system-wide
ls /usr/share/fonts/

# Check per-user fonts
ls ~/.local/share/fonts/
```

There's also `/usr/local/share/fonts/` for fonts installed manually outside the package manager — useful for fonts you've acquired independently and want available system-wide without package management involvement.

### Font Formats: What Actually Matters

**TrueType (TTF)** is the workhorse format. It's universally supported, renders well at both screen and print sizes, and is what you'll encounter most often. When in doubt, prefer TTF.

**OpenType (OTF)** is TTF's more capable successor. It supports more advanced typographic features — ligatures, alternate glyphs, advanced kerning — and can embed either TrueType or PostScript outlines. For display and UI work, TTF and OTF are functionally equivalent; OTF shines in professional typography.

**WOFF/WOFF2** are web-optimized, compressed font formats. They're designed for browsers and are generally not useful for desktop font configuration. You can't just drop a WOFF file in `~/.local/share/fonts/` and expect it to work reliably in GTK applications.

**Type 1 (PostScript)** fonts are legacy PostScript-based fonts from Adobe. FreeType still renders them, but support is being phased out — modern fontconfig and FreeType deprecated full Type 1 support around 2022-2023. Avoid relying on them for new setups.

**Bitmap fonts** (BDF, PCF) are pixel-based rather than vector. They look sharp at their native pixel sizes but don't scale. They're still used in certain terminal emulators (Terminus is a beloved example) but are generally not appropriate for modern HiDPI displays or general-purpose use.

### Installing Fonts

**Via package manager** is the cleanest approach for well-known font families:

```bash
# Debian/Ubuntu — install the Noto family and Liberation fonts
sudo apt install fonts-noto fonts-liberation

# Arch Linux
sudo pacman -S noto-fonts noto-fonts-cjk noto-fonts-emoji ttf-liberation

# Fedora/RHEL
sudo dnf install google-noto-fonts-common liberation-fonts
```

**Manually** is straightforward: copy the font files to the appropriate directory and rebuild the cache.

```bash
# Install a font for the current user only
mkdir -p ~/.local/share/fonts/MyFont
cp MyFont-Regular.ttf MyFont-Bold.ttf ~/.local/share/fonts/MyFont/

# Rebuild the font cache
fc-cache -fv
```

The `-f` flag forces a full rebuild even if timestamps suggest nothing changed. The `-v` flag gives verbose output so you can confirm your directories were scanned.

### Refreshing the Font Cache

fontconfig caches font metadata in `~/.cache/fontconfig/` (user) and `/var/cache/fontconfig/` (system). After installing or removing fonts, the cache needs to be updated:

```bash
# Rebuild user cache
fc-cache -fv

# Rebuild system cache (requires root)
sudo fc-cache -fv

# Nuke the user cache entirely and rebuild from scratch
rm -rf ~/.cache/fontconfig && fc-cache -fv
```

The last option is useful when you're chasing a ghost — stale cache entries can cause fonts to appear available when they're not, or persist after deletion.

## 2. Fontconfig Deep Dive

### What Fontconfig Is and Why It Exists

fontconfig is the library responsible for font discovery, configuration, and matching on Linux (and other Unix-like systems). It answers a deceptively complex question: _when application X requests a font named "sans-serif" at 12pt for Japanese text, which font file should actually be used?_

Without fontconfig, every application would need its own font discovery logic. With it, there's a single, configurable layer that all GTK, Qt, and most other applications delegate to. The official documentation lives at [https://www.freedesktop.org/software/fontconfig/fontconfig-user.html](https://www.freedesktop.org/software/fontconfig/fontconfig-user.html).

### The Configuration Hierarchy

fontconfig processes configuration files in a specific order, with later rules generally winning over earlier ones (subject to edit priorities, which we'll cover).

`/etc/fonts/fonts.conf` is the root configuration file generated by your distribution. **Do not edit this file directly.** It will be overwritten on package upgrades, and your changes will vanish.

`/etc/fonts/conf.d/` contains numbered snippet files (e.g., `10-hinting-slight.conf`, `65-nonlatin.conf`) that the system `fonts.conf` includes. These are often symlinks from `/etc/fonts/conf.avail/`. You can add system-wide customizations here, but per-user configuration is preferred for anything that isn't system policy.

`~/.config/fontconfig/fonts.conf` is where you should put your personal font configuration. This file is loaded after all system configuration, giving your preferences the highest priority. This is the preferred approach for customizing rendering, setting default fonts, or configuring emoji.

`~/.config/fontconfig/conf.d/` works the same as the system equivalent — you can place numbered snippet files here to organize your configuration logically.

### The XML Configuration Structure

fontconfig configs are XML files. The root element is always `<fontconfig>`. Here's a minimal skeleton:

```xml
<?xml version="1.0"?>
<!DOCTYPE fontconfig SYSTEM "fonts.dtd">
<fontconfig>
  <!-- your rules go here -->
</fontconfig>
```

The key building blocks you need to understand are:

**`<match>`** is the workhorse element. It specifies a pattern to match against font requests and a set of edits to apply. The `target` attribute controls what's being matched — `target="pattern"` matches incoming font requests (the most common case), while `target="font"` matches after a font has been selected (useful for rendering tweaks like hinting).

**`<test>`** appears inside `<match>` to specify conditions. You test a named property (like `family`, `lang`, `spacing`) against a value using a `compare` attribute (`eq`, `contains`, `not_eq`, etc.).

**`<edit>`** also appears inside `<match>` and specifies what to do when the test passes. The `mode` attribute controls how the edit applies — `prepend` adds to the front of a list, `append` adds to the end, and `assign` replaces the value entirely.

**`<alias>`**, **`<family>`**, **`<prefer>`**, **`<accept>`**, and **`<default>`** provide a higher-level shorthand for the common case of setting up font substitution chains without writing raw `<match>` blocks.

### Font Substitution and Fallback Chains

When an application requests a font, fontconfig builds an ordered list of candidates. If the first font can't render a particular character, fontconfig falls through to the next candidate. This fallback chain is how you get sensible behavior when, say, a Latin text font encounters a CJK ideograph — fontconfig falls back to a CJK-capable font rather than rendering boxes.

You control this chain by prepending or appending to the `family` property in a font pattern. Prepending means "try this font first"; appending means "try this font as a last resort."

### Priority Order

fontconfig processes rules in document order within a config file, and processes config files in sorted order within `conf.d/` directories. Among multiple `<edit>` operations on the same property:

- `prepend_first` / `prepend` push values to the front of the candidate list.
- `append_last` / `append` push values to the back.
- `assign` replaces entirely.

User configuration (`~/.config/fontconfig/`) loads after system configuration, so user-level rules applying with the same mode will effectively win. However, a system rule using `assign` can wipe out what user rules later try to prepend — be aware of this when debugging unexpected font selections.

## 3. Essential Fontconfig Use Cases

### Setting Default Sans-Serif, Serif, and Monospace Fonts

This is the single most useful thing you'll put in your `fonts.conf`. The generic families `sans-serif`, `serif`, and `monospace` are what applications request when they don't specify a concrete font. Mapping them to your preferred fonts is done with `<alias>`:

```xml
<fontconfig>
  <!-- Default sans-serif: prefer Inter, fall back to Noto Sans -->
  <alias>
    <family>sans-serif</family>
    <prefer>
      <family>Inter</family>
      <family>Noto Sans</family>
    </prefer>
  </alias>

  <!-- Default serif: prefer Source Serif 4 -->
  <alias>
    <family>serif</family>
    <prefer>
      <family>Source Serif 4</family>
      <family>Noto Serif</family>
    </prefer>
  </alias>

  <!-- Default monospace: prefer JetBrains Mono -->
  <alias>
    <family>monospace</family>
    <prefer>
      <family>JetBrains Mono</family>
      <family>Noto Sans Mono</family>
    </prefer>
  </alias>
</fontconfig>
```

### Font Substitution: Replacing One Font with Another

If a legacy application hard-codes "Helvetica" or a document requests "Arial" and you'd rather serve Inter, use a `<match>` substitution:

```xml
<!-- Map requests for Helvetica or Arial to Inter -->
<match target="pattern">
  <test qual="any" name="family">
    <string>Helvetica</string>
  </test>
  <edit name="family" mode="assign" binding="same">
    <string>Inter</string>
  </edit>
</match>

<match target="pattern">
  <test qual="any" name="family">
    <string>Arial</string>
  </test>
  <edit name="family" mode="assign" binding="same">
    <string>Inter</string>
  </edit>
</match>
```

The `binding="same"` attribute tells fontconfig to treat the substituted font as equivalent in strength to the original request, preserving priority behavior correctly.

### Language-Specific Font Preferences

For CJK (Chinese, Japanese, Korean) text, you almost always want a dedicated CJK font rather than falling through to a Latin font that happens to have a few CJK glyphs. Noto Sans CJK is the gold standard here:

```xml
<!-- Prefer Noto Sans CJK for Chinese, Japanese, and Korean -->
<match target="pattern">
  <test name="lang" compare="contains">
    <string>zh</string>
  </test>
  <edit name="family" mode="prepend">
    <string>Noto Sans CJK SC</string>
  </edit>
</match>

<match target="pattern">
  <test name="lang" compare="contains">
    <string>ja</string>
  </test>
  <edit name="family" mode="prepend">
    <string>Noto Sans CJK JP</string>
  </edit>
</match>

<match target="pattern">
  <test name="lang" compare="contains">
    <string>ko</string>
  </test>
  <edit name="family" mode="prepend">
    <string>Noto Sans CJK KR</string>
  </edit>
</match>
```

### Disabling Bitmap Fonts Globally

Bitmap fonts can look out of place on modern displays and often interfere with font matching in unexpected ways. Disabling them globally is safe for most modern setups:

```xml
<!-- Reject all bitmap (non-scalable) fonts -->
<selectfont>
  <rejectfont>
    <pattern>
      <patelt name="scalable">
        <bool>false</bool>
      </patelt>
    </pattern>
  </rejectfont>
</selectfont>
```

### Configuring Hinting and Anti-Aliasing

Font rendering quality depends on hinting (adjusting glyph outlines to fit the pixel grid) and anti-aliasing (smoothing edges). For LCD screens, subpixel rendering with an appropriate LCD filter gives the best results. Apply these with `target="font"` matches so they affect the rendered font, not the selection pattern:

```xml
<!-- Enable anti-aliasing globally -->
<match target="font">
  <edit name="antialias" mode="assign">
    <bool>true</bool>
  </edit>
</match>

<!-- Enable slight hinting (good balance of sharpness vs. fidelity) -->
<match target="font">
  <edit name="hinting" mode="assign">
    <bool>true</bool>
  </edit>
  <edit name="hintstyle" mode="assign">
    <const>hintslight</const>
  </edit>
</match>

<!-- Enable subpixel rendering for LCD screens (RGB layout is most common) -->
<match target="font">
  <edit name="rgba" mode="assign">
    <const>rgb</const>
  </edit>
  <edit name="lcdfilter" mode="assign">
    <const>lcddefault</const>
  </edit>
</match>
```

Valid `hintstyle` values are `hintnone`, `hintslight`, `hintmedium`, and `hintfull`. For most text, `hintslight` is the right call — it improves sharpness without distorting letter shapes as aggressively as `hintfull`. On HiDPI displays (2x+), hinting matters less because pixels are small enough that you gain little from grid-fitting.

## 4. Fontconfig CLI Tools

These tools are your primary interface for understanding and debugging fontconfig's behavior. Learn them; they save enormous time.

### `fc-list` — Enumerate Available Fonts

```bash
# List all fonts (family name and file)
fc-list

# Filter by family
fc-list | grep -i "noto"

# List fonts supporting a specific language
fc-list :lang=ja

# Show family, style, and file for monospace fonts
fc-list :spacing=mono family style file
```

Example output from `fc-list :lang=ja`:

```
/usr/share/fonts/noto-cjk/NotoSansCJK-Regular.ttc: Noto Sans CJK JP:style=Regular
/usr/share/fonts/noto-cjk/NotoSerifCJK-Regular.ttc: Noto Serif CJK JP:style=Regular
```

### `fc-match` — Show Which Font Wins

This is the most important debugging tool. It shows exactly which font fontconfig would select for a given request, after applying all your configuration rules:

```bash
# Which font is used for generic sans-serif?
fc-match sans-serif
# DejaVu Sans:style=Book   (or whatever your config maps it to)

# Which font for a specific family?
fc-match "Inter"
# Inter:style=Regular

# Verbose output — shows the full sorted candidate list
fc-match -s sans-serif | head -10

# Test with language parameter
fc-match "sans-serif:lang=ja"
# NotoSansCJKjp-Regular.otf: "Noto Sans CJK JP" "Regular"
```

If `fc-match` shows a different font than you expect, your config rules aren't firing as intended — use this as your first diagnostic step.

### `fc-query` — Inspect a Font File

```bash
# Show all metadata for a font file
fc-query /usr/share/fonts/truetype/liberation/LiberationSans-Regular.ttf

# Extract just the family name
fc-query --format="%{family}\n" MyFont.ttf
```

This is useful when you've downloaded a font and need to know the exact family name string to use in your fontconfig XML — the family name in the file metadata is what fontconfig uses, not the filename.

### `fc-scan` — Scan a Directory

```bash
# Scan a directory and print font metadata without adding to cache
fc-scan ~/.local/share/fonts/
```

Useful for verifying that fonts in a new directory are readable and have the properties you expect before rebuilding the cache.

### `fc-cache` — Rebuild the Cache

```bash
# Force full rebuild, verbose
fc-cache -fv

# Rebuild system-wide cache
sudo fc-cache -fv

# Dry run — show what would be done
fc-cache -v --dry-run  # (not all versions support this)
```

### `FC_DEBUG` — Deep Debugging

The `FC_DEBUG` environment variable enables detailed logging of fontconfig's internal decision-making. It takes a bitmask of debug flags:

```bash
# FC_DEBUG=1: match logging
# FC_DEBUG=4: font scanning
# FC_DEBUG=256: config loading

# See exactly why fontconfig chose the font it did for a sans-serif request
FC_DEBUG=1 fc-match sans-serif 2>&1 | less

# Debug config file loading
FC_DEBUG=256 fc-match monospace 2>&1 | grep -i "loading"
```

The output is verbose, but when `fc-match` gives you a surprising result and you can't figure out why, `FC_DEBUG=1` will show you every rule that was evaluated and what it did.

## 5. Emoji on Linux

### Why Emoji Rendering Is Fragmented

Emoji rendering on Linux is genuinely messy, and understanding why helps you fix it. The core issue is that emoji are Unicode codepoints, and _many_ fonts include those codepoints — not just dedicated emoji fonts. Your regular sans-serif font might include basic smiley faces as monochrome glyphs. When fontconfig matches a font for text that happens to contain emoji, it often reaches your text font first, yielding ugly monochrome glyphs instead of colorful emoji — or boxes if your text font doesn't have those codepoints at all.

The second issue is that color emoji fonts use different rendering mechanisms (Microsoft's COLR/CPAL tables, Google's CBDT/CBLC bitmap layers, or SVG via OpenType SVG) and different applications support different subsets of these formats.

### Main Emoji Font Options

**Noto Emoji** from Google comes in two variants. The monochrome `fonts-noto-mono` package is a traditional outline font — good for terminal use, not for colorful emoji. The color variant (`fonts-noto-color-emoji` on Debian/Ubuntu) uses CBDT/CBLC bitmap layers and renders proper color emoji in most GTK applications.

**Twemoji** (from Twitter/X, now maintained by the community) is available via various packaging routes. It uses SVG-based color rendering. It's aesthetically distinctive and popular for consistent cross-platform appearance.

**JoyPixels** is a commercial emoji set with a free tier. Less common on Linux desktops but available.

**Segoe UI Emoji** (Windows) can technically be copied from a Windows installation and used on Linux, but this is a licensing grey area — Microsoft's font EULA doesn't permit redistribution. Use it for personal setups at your own discretion.

For most users, **Noto Color Emoji is the best default choice** — it's well-maintained, broadly compatible, covers the full Unicode emoji range, and is available in every major distro's package repositories.

### Configuring Fontconfig for Emoji Fallback

The key insight is that you need to append your emoji font to the fallback chain for every generic family, so that when a text font can't render an emoji codepoint, the emoji font gets called up. You also need to make sure your emoji font is preferred _over_ text fonts for codepoints in the emoji ranges.

```xml
<!-- Append emoji font to all generic families as fallback -->
<match target="pattern">
  <test qual="any" name="family">
    <string>sans-serif</string>
  </test>
  <edit name="family" mode="append_last">
    <string>Noto Color Emoji</string>
  </edit>
</match>

<match target="pattern">
  <test qual="any" name="family">
    <string>serif</string>
  </test>
  <edit name="family" mode="append_last">
    <string>Noto Color Emoji</string>
  </edit>
</match>

<match target="pattern">
  <test qual="any" name="family">
    <string>monospace</string>
  </test>
  <edit name="family" mode="append_last">
    <string>Noto Color Emoji</string>
  </edit>
</match>
```

### Preventing Text Fonts from Stealing Emoji Codepoints

The more subtle problem is when your text font _does_ contain emoji codepoints but renders them poorly (usually as monochrome outlines). The fix is to block specific codepoint ranges from being matched by non-emoji fonts, using fontconfig's `charset` matching. This is done with a `<match target="font">` rule:

```xml
<!-- Prevent non-emoji fonts from matching emoji codepoints -->
<!-- This tells fontconfig to remove emoji ranges from non-emoji fonts' charsets -->
<match target="font">
  <test name="family" compare="not_eq">
    <string>Noto Color Emoji</string>
  </test>
  <edit name="charset" mode="assign">
    <minus>
      <name>charset</name>
      <charset>
        <!-- Emoticons block -->
        <range>
          <int>0x1F600</int>
          <int>0x1F64F</int>
        </range>
        <!-- Misc Symbols and Pictographs -->
        <range>
          <int>0x1F300</int>
          <int>0x1F5FF</int>
        </range>
        <!-- Transport and Map Symbols -->
        <range>
          <int>0x1F680</int>
          <int>0x1F6FF</int>
        </range>
        <!-- Supplemental Symbols and Pictographs -->
        <range>
          <int>0x1F900</int>
          <int>0x1F9FF</int>
        </range>
      </charset>
    </minus>
  </edit>
</match>
```

This rule removes emoji codepoints from every font's reported charset _except_ Noto Color Emoji, forcing fontconfig to select the emoji font for those characters.

## 6. Apple Emoji on Linux Using `apple-emoji-ttf`

### What It Is and Why You Might Want It

[`apple-emoji-ttf`](https://github.com/samuelngs/apple-emoji-ttf) is a community project that packages Apple's "Apple Color Emoji" font (extracted from macOS/iOS) as a standard TTF file compatible with Linux font rendering. Apple's emoji set is widely considered the most aesthetically refined of the major emoji collections — many users who've grown accustomed to the Apple emoji style on other platforms want to replicate it on their Linux desktop.

The font uses Apple's proprietary sbix (Standard Bitmap Graphics) color format, which is supported by FreeType and renders correctly in most modern applications.

### Step-by-Step Installation

**Step 1: Download the TTF from the releases page.**

Visit [https://github.com/samuelngs/apple-emoji-ttf/releases](https://github.com/samuelngs/apple-emoji-ttf/releases) and download the latest `AppleColorEmoji.ttf` release.

**Step 2: Install to `~/.local/share/fonts/`.**

```bash
mkdir -p ~/.local/share/fonts/apple-emoji
cp AppleColorEmoji.ttf ~/.local/share/fonts/apple-emoji/
```

**Step 3: Rebuild the font cache.**

```bash
fc-cache -fv
```

You'll see output confirming that your `~/.local/share/fonts/apple-emoji/` directory was scanned.

**Step 4: Verify the font is available.**

```bash
fc-list | grep -i apple
# Should output something like:
# /home/youruser/.local/share/fonts/apple-emoji/AppleColorEmoji.ttf: Apple Color Emoji:style=Regular
```

### Complete Fontconfig Configuration for Apple Color Emoji

This snippet sets Apple Color Emoji as your system emoji font, registers it as a fallback for all generic families, and suppresses competing emoji codepoints in other fonts:

```xml
<?xml version="1.0"?>
<!DOCTYPE fontconfig SYSTEM "fonts.dtd">
<fontconfig>

  <!-- Register "Apple Color Emoji" as the canonical emoji font -->
  <alias>
    <family>emoji</family>
    <prefer>
      <family>Apple Color Emoji</family>
    </prefer>
  </alias>

  <!-- Append Apple Color Emoji to all generic families -->
  <match target="pattern">
    <test qual="any" name="family">
      <string>sans-serif</string>
    </test>
    <edit name="family" mode="append_last">
      <string>Apple Color Emoji</string>
    </edit>
  </match>

  <match target="pattern">
    <test qual="any" name="family">
      <string>serif</string>
    </test>
    <edit name="family" mode="append_last">
      <string>Apple Color Emoji</string>
    </edit>
  </match>

  <match target="pattern">
    <test qual="any" name="family">
      <string>monospace</string>
    </test>
    <edit name="family" mode="append_last">
      <string>Apple Color Emoji</string>
    </edit>
  </match>

  <!-- Prevent system emoji fonts (e.g., Noto Color Emoji) from competing -->
  <!-- If you have Noto Color Emoji installed, deprioritize it -->
  <match target="pattern">
    <test qual="any" name="family">
      <string>Noto Color Emoji</string>
    </test>
    <edit name="family" mode="assign" binding="weak">
      <string>Apple Color Emoji</string>
    </edit>
  </match>

  <!-- Strip emoji codepoints from non-emoji fonts -->
  <match target="font">
    <test name="family" compare="not_eq">
      <string>Apple Color Emoji</string>
    </test>
    <edit name="charset" mode="assign">
      <minus>
        <name>charset</name>
        <charset>
          <range><int>0x1F600</int><int>0x1F64F</int></range>
          <range><int>0x1F300</int><int>0x1F5FF</int></range>
          <range><int>0x1F680</int><int>0x1F6FF</int></range>
          <range><int>0x1F900</int><int>0x1F9FF</int></range>
          <range><int>0x2600</int><int>0x26FF</int></range>
          <range><int>0x2700</int><int>0x27BF</int></range>
        </charset>
      </minus>
    </edit>
  </match>

</fontconfig>
```

Save this as `~/.config/fontconfig/conf.d/99-apple-emoji.conf` and rebuild the cache.

### Testing That Emoji Render Correctly

### Caveats: Licensing and When NOT to Use This

Apple Color Emoji is Apple's proprietary intellectual property. The font is distributed with macOS and iOS under Apple's standard software license, which prohibits redistribution. The `apple-emoji-ttf` project exists in a legal grey area — it repackages the font for easier use, but does not have Apple's permission to redistribute it.

**When you should not use this setup:** In any professional, commercial, or shared environment. If you're configuring a server, a CI/CD environment, a shared workstation, or any system where others might access or redistribute the font, use Noto Color Emoji instead. It's licensed under the SIL Open Font License and is entirely free for any use.

For personal, single-user desktop setups where you've legitimately accessed the font from your own Apple device, the practical risk is low — but be clear-eyed about the legal situation and never redistribute the font file.

## 7. Putting It All Together

### A Complete, Annotated `~/.config/fontconfig/fonts.conf`

This is a production-ready configuration combining custom defaults, CJK language support, rendering tuning, and Apple Color Emoji setup:

```xml
<?xml version="1.0"?>
<!DOCTYPE fontconfig SYSTEM "fonts.dtd">
<fontconfig>

  <!-- ========================================================
       RENDERING QUALITY
       ======================================================== -->

  <!-- Enable anti-aliasing for all fonts -->
  <match target="font">
    <edit name="antialias" mode="assign"><bool>true</bool></edit>
  </match>

  <!-- Slight hinting: sharpens without distorting letterforms.
       Use hintfull on low-DPI displays if you want crisper text. -->
  <match target="font">
    <edit name="hinting" mode="assign"><bool>true</bool></edit>
    <edit name="hintstyle" mode="assign"><const>hintslight</const></edit>
  </match>

  <!-- Subpixel rendering for RGB LCD screens.
       Change to "bgr" if your screen uses BGR subpixel layout. -->
  <match target="font">
    <edit name="rgba" mode="assign"><const>rgb</const></edit>
    <edit name="lcdfilter" mode="assign"><const>lcddefault</const></edit>
  </match>

  <!-- Disable bitmap fonts system-wide -->
  <selectfont>
    <rejectfont>
      <pattern>
        <patelt name="scalable"><bool>false</bool></patelt>
      </pattern>
    </rejectfont>
  </selectfont>


  <!-- ========================================================
       DEFAULT FONT FAMILIES
       Change Inter/Source Serif 4/JetBrains Mono to your
       preferred fonts. Install them first!
       ======================================================== -->

  <alias>
    <family>sans-serif</family>
    <prefer>
      <family>Inter</family>

    </prefer>

    <prefer>
      <family>JetBrains Mono</family>
      <family>Noto Sans Mono</family>
      <family>DejaVu Sans Mono</family>
    </prefer>
  </alias>


  <!-- ========================================================
       FONT SUBSTITUTIONS
       Map legacy/unavailable font names to modern equivalents
       ======================================================== -->

  <match target="pattern">
    <test qual="any" name="family"><string>Helvetica</string></test>
    <edit name="family" mode="assign" binding="same"><string>Inter</string></edit>
  </match>

  <match target="pattern">
    <test qual="any" name="family"><string>Arial</string></test>
    <edit name="family" mode="assign" binding="same"><string>Inter</string></edit>

  <match target="pattern">

  </alias>

  <!-- Append emoji to generic families as final fallback -->
  <match target="pattern">
    <test qual="any" name="family"><string>sans-serif</string></test>
```

### Debugging Font Issues by Application Type

Different application stacks use fontconfig differently, and a font issue in one app may not appear in another.

**GTK applications** (Nautilus, GNOME Text Editor, Inkscape) are the most fontconfig-native. If `fc-match` shows the right font but a GTK app renders differently, check if the app overrides fonts in its own settings (GNOME's Settings → Appearance, or `~/.config/gtk-3.0/settings.ini`). The key GTK font settings are `gtk-font-name` (GTK3) and the equivalent in GTK4 gsettings.

```bash
# Check GTK3 font setting via gsettings
gsettings get org.gnome.desktop.interface font-name
gsettings set org.gnome.desktop.interface font-name "Inter 11"
```

**Qt applications** (Dolphin, Konsole, VLC) use fontconfig for discovery but apply their own rendering pipeline. Qt's font settings are configurable via `qt5ct` or `qt6ct` (install the relevant package), or through KDE System Settings if you're on Plasma. If Qt apps look different from GTK apps, this is usually why.

**Terminal emulators** vary enormously. Kitty, Alacritty, and WezTerm have their own font configuration in their respective config files and don't always respect fontconfig for their primary font — but they do fall back to it for codepoints not in their configured font. Emoji rendering in terminals specifically depends on whether the emulator supports the color emoji format of your chosen font (CBDT, COLR, sbix).

**Browsers** are a world unto themselves. Firefox respects fontconfig reasonably well for body text but makes its own choices for UI elements. You can set fonts in `about:preferences#general`. Chromium-based browsers (Chrome, Brave, Edge on Linux) often use their own font matching logic and may bypass parts of fontconfig. If emoji look wrong in Chrome but correct in Firefox, suspect Chromium's independent handling rather than your fontconfig config.

### GUI Font Management with `font-manager`

For casual font browsing, preview, and organization, `font-manager` is excellent:

```bash
# Ubuntu/Debian
sudo apt install font-manager

# Arch
sudo pacman -S font-manager
```

It provides a clean GUI to preview installed fonts, enable/disable individual families, install new fonts by drag-and-drop, and browse fonts by category and language coverage. It's not a replacement for understanding fontconfig — when something goes wrong, you still need the CLI tools — but it's a pleasant way to explore what you have installed.

## 8. The conf.d Numbering System: Load Order, Priority, and Overrides

This is one of the most misunderstood aspects of fontconfig, and getting it wrong leads to hours of frustrated debugging where your carefully written rules seem to have no effect.

### Files Are Loaded in Strict Lexicographic Order

When fontconfig processes a `conf.d/` directory, it loads every file matching the pattern `[0-9][0-9]*.conf` in sorted (lexicographic) filename order. `01-foo.conf` is loaded before `10-bar.conf`, which is loaded before `99-overrides.conf`. This is true for both `/etc/fonts/conf.d/` and `~/.config/fontconfig/conf.d/`.

The critical implication: **a rule in `01-foo.conf` is processed before a rule in `99-bar.conf`.** Whether that means the earlier or later rule "wins" depends entirely on which `mode` the edit uses — and this is where most confusion originates.

### The Official Numbering Convention

The files are loaded in numeric order, and the structure of the configuration has led to the following conventions in usage:

| Range   | Purpose                                               |
| ------- | ----------------------------------------------------- |
| `00–09` | Font directory declarations                           |
| `10–19` | System rendering defaults (antialiasing, hinting)     |
| `20–29` | Font rendering options                                |
| `30–39` | Family substitution rules                             |
| `40–49` | Generic identification: map family → generic          |
| `50–59` | Alternate config file loading (including user config) |
| `60–69` | Generic aliases: map generic → family                 |
| `70–79` | Select font (adjust which fonts are available)        |
| `80–89` | `match target="scan"` rules (modify scanned patterns) |
| `90–99` | Font synthesis rules                                  |

This isn't just organizational convention — it reflects the order in which fontconfig logically needs to process things. You can't map generic names to families (60s range) before you've established what the generic names are (40s range). You can't do font synthesis (90s range) before you've finalized which font was selected (70s range).

### The 50-user.conf Gateway: Where Your Config Enters

Here's the part the Arch Wiki flags as frequently misunderstood: user rules defined in `$XDG_CONFIG_HOME/fontconfig/fonts.conf` and in the directory `$XDG_CONFIG_HOME/fontconfig/conf.d` are loaded via the file `/etc/fonts/conf.d/50-user.conf` and typically take precedence over rules defined in files starting with a higher number.

This means your personal `~/.config/fontconfig/fonts.conf` is essentially injected into the system config pipeline at position 50 — after rendering defaults (10s–20s) and family substitutions (30s–40s), but before generic aliasing (60s) and font selection (70s). Rules in system files numbered 60 and above are processed _after_ your user config.

The practical upshot: **a `prepend` rule you write in your user `fonts.conf` will take effect before the system's `65-nonlatin.conf` or `69-unifont.conf`**. But if you need to override something in `30-metric-aliases.conf` (a 30s-range system file), your user config lands too late in the pipeline to override it cleanly with a `prepend` — you'd need to place your rule in `~/.config/fontconfig/conf.d/20-my-override.conf` so it's injected at the right position.

You can verify exactly when your files are loaded with:

```bash
FC_DEBUG=1024 fc-match sans-serif 2>&1 | grep -i "loading\|scanning"
```

### Prepend vs. Append: The Priority Inversion That Trips Everyone Up

This is the subtlest part of the numbering system. Fontconfig will sort the ruleset files in `/etc/fonts/conf.d/` by filename before evaluating them. Usually that means the files with the smaller prefix will have higher precedence. Unfortunately the fontconfig syntax is pretty flexible and allows prepending effects to existing rules.

The key point is that the `mode` of your edit determines whether numerically earlier or later files "win":

**For `mode="append"` rules:** the last file wins. A `99-` file appending to the family list wins over a `10-` file that also appended — the 99 entry ends up further back in the list.

**For `mode="prepend"` rules:** this inverts. A `99-` file prepending to the family list inserts _before_ entries added by a `10-` file that already ran, which means the `99-` entry ends up at the front. So for `prepend` rules, higher numbers have higher priority. Since it uses the prepend mode, it follows an inverted priority: rulesets with high numeral prefixes take precedence over rulesets with low numeral prefixes.

This is counterintuitive and the source of endless confusion. The rule of thumb:

- If you're using `mode="append"` or `mode="append_last"` and want your rule to be the final word, use a high number like `90-` or `99-`.
- If you're using `mode="prepend"` and want your font to be the first candidate, also use a high number like `90-` or `99-` — because later prepends end up in front of earlier ones.
- If you want to establish a baseline that others can override, use a low number.

### The conf.avail / conf.d Symlink Pattern

Most distributions store the full set of available fontconfig snippets in `/usr/share/fontconfig/conf.avail/` (or `/etc/fonts/conf.avail/` on some distros) and only _symlink_ the ones they want active into `/etc/fonts/conf.d/`. This means you can enable and disable system-level font configuration snippets without editing or deleting files:

```bash
# See what's available but not active
ls /usr/share/fontconfig/conf.avail/

# Enable a snippet (e.g., disable embedded bitmaps in fonts)
sudo ln -s /usr/share/fontconfig/conf.avail/70-no-bitmaps.conf /etc/fonts/conf.d/

# Disable a snippet that's currently active
sudo rm /etc/fonts/conf.d/10-hinting-slight.conf  # remove the symlink, not the source

# Rebuild cache after changes
sudo fc-cache -fv
```

On Debian/Ubuntu, the `dpkg-reconfigure fontconfig` command manages some of these symlinks through the package system — this is how toggling subpixel rendering in the Debian fontconfig package postinst script works.

### A Concrete Load Order Example

To make this concrete, here's the order fontconfig processes things on a typical Arch Linux system when you run `fc-match sans-serif`:

```
/etc/fonts/fonts.conf              ← root config, includes conf.d/
  /etc/fonts/conf.d/10-hinting-slight.conf    ← hinting defaults
  /etc/fonts/conf.d/10-scale-bitmap-fonts.conf
  /etc/fonts/conf.d/20-unhint-small-vera.conf
  /etc/fonts/conf.d/30-metric-aliases.conf    ← Arial→Liberation etc.
  /etc/fonts/conf.d/45-latin.conf             ← Latin family identification
  /etc/fonts/conf.d/49-sansserif.conf         ← add sans-serif if no generic
  /etc/fonts/conf.d/50-user.conf              ← *** YOUR CONFIG LOADS HERE ***
    ~/.config/fontconfig/fonts.conf
    ~/.config/fontconfig/conf.d/99-emoji.conf
  /etc/fonts/conf.d/60-latin.conf             ← generic→family aliases
  /etc/fonts/conf.d/65-nonlatin.conf
  /etc/fonts/conf.d/70-no-bitmaps.conf        ← reject bitmap fonts
```

Anything you write in your user config at position 50 runs between the 49s and 60s ranges. Rules in your user conf.d snippets are processed at position 50 regardless of what number prefix you give them — the numbers on your user snippets only control ordering _relative to each other_, not relative to system files.

## 9. Firefox and Fonts: Why about:config Is Not Optional

Firefox on Linux has its own internal font matching pipeline that partially bypasses the system fontconfig. This is one of the most common sources of "my fonts look great everywhere except Firefox" complaints, and it's entirely fixable — once you know where to look.

### Why Firefox Doesn't Fully Respect fontconfig

Firefox implements its own font enumeration and selection logic rather than calling directly into the fontconfig library for every font decision. It uses fontconfig to _discover_ fonts, but then applies its own CSS font-matching algorithm on top. This means:

- Firefox may cap how many fontconfig substitution rules it applies.
- Firefox ships with bundled fonts (notably Twemoji Mozilla) that can override your system emoji config.
- Firefox's per-family font preferences, set via `about:preferences`, are stored internally and take precedence over fontconfig aliases.

### Accessing about:config

Navigate to `about:config` in the address bar and accept the warning. Use the search bar at the top to find the settings below. To change a string value, double-click the entry and type your value. To change a boolean, click the toggle icon on the right.

### Essential Font Settings in about:config

**`gfx.font_rendering.fontconfig.max_generic_substitutions`**

This is the single most impactful setting for fontconfig users. Firefox has a setting which determines how many replacements it will allow from fontconfig. To allow it to use all your replacement rules, change `gfx.font_rendering.fontconfig.max_generic_substitutions` to 127 (the highest possible value). The default is somewhere around 3–5 on most versions, which means if you have a long fallback chain (sans-serif → Inter → Noto Sans → Noto Sans CJK → Apple Color Emoji), Firefox silently truncates it.

Set this to `127` — it's the maximum and there's no performance downside for typical usage.

**`font.name-list.sans-serif` / `font.name-list.serif` / `font.name-list.monospace`**

These are Firefox's internal per-family font preferences, organized by script/language. They're split by script, so you'll see entries like `font.name-list.sans-serif.x-western`, `font.name-list.sans-serif.zh-CN`, `font.name-list.sans-serif.ja`, etc.

To set your preferred sans-serif for Western text:

1. Search for `font.name-list.sans-serif.x-western`
2. If it doesn't exist, click the `+` button to create a new string preference
3. Set the value to a comma-separated list: `Inter, Noto Sans, DejaVu Sans`

Firefox tries fonts in this list from left to right, falling back to fontconfig for anything not listed. Setting this explicitly is the reliable way to get a specific font in Firefox regardless of what your fontconfig is doing.

**`font.name-list.emoji`**

To use the system emoji font, set `font.name-list.emoji` to `emoji` in about:config. The value `emoji` tells Firefox to use whatever fontconfig resolves for the generic `emoji` family — which, if you've followed the fontconfig configuration in earlier sections, will be Apple Color Emoji or Noto Color Emoji as you've configured it.

**`gfx.font_rendering.opentype_svg.enabled`**

Firefox ships with the Twemoji Mozilla font. To prevent the Mozilla font interfering with your system emoji font, change `gfx.font_rendering.opentype_svg.enabled` to `false`, or remove `/usr/lib/firefox/fonts/TwemojiMozilla.ttf`.

Setting this to `false` disables Firefox's OpenType SVG font rendering, which is specifically what the bundled Twemoji Mozilla uses. Without it, Firefox will use Twemoji for emoji even when you've configured your system to use something else. Deleting the font file is more permanent and doesn't affect anything else, but the `about:config` toggle is cleaner and survives Firefox updates.

### The Full Firefox Font Checklist

Here's the sequence to follow when Firefox fonts aren't behaving:

```
1. Set gfx.font_rendering.fontconfig.max_generic_substitutions = 127
2. Set font.name-list.emoji = emoji
3. Set gfx.font_rendering.opentype_svg.enabled = false  (if using non-Twemoji emoji)
4. Set font.name-list.sans-serif.x-western = Your Font, Fallback Font
5. Set font.name-list.monospace.x-western = Your Mono Font
6. Restart Firefox completely (not just a new tab)
7. Verify with: View Page Source → check what font is rendering via browser DevTools
```

### Checking Font Rendering in Firefox DevTools

The browser DevTools inspector has a "Fonts" panel that tells you exactly which font is being used to render selected text — this is more reliable than guessing from visual appearance.

1. Open DevTools with `F12`
2. In the Inspector tab, click on any text element on the page
3. Click the "Fonts" tab in the right panel (you may need to expand it)
4. Firefox will show you the exact font file being used, including the path

This is the equivalent of `fc-match` for the browser context and should be your first stop when Firefox is using an unexpected font.

### Overriding Fonts Per-Application with FONTCONFIG_FILE

For any application — not just Firefox — you can override the fontconfig config file entirely by setting the `FONTCONFIG_FILE` environment variable before launching:

```bash
# Launch Firefox with a custom fontconfig config
FONTCONFIG_FILE=~/.config/fontconfig/firefox-fonts.conf firefox &
```

This is useful when an application behaves oddly with your global config and you want to test whether a minimal config reproduces the problem, or if you want a completely different font profile for a specific app without affecting everything else. `FONTCONFIG_PATH` similarly overrides which directory fontconfig scans for config files.

### A Note on Chromium and Electron Apps

Chromium-based browsers (Chrome, Brave, Edge, Vivaldi) and Electron applications largely ignore fontconfig for font selection, using their own Skia-based font fallback logic. You cannot meaningfully configure fonts in these applications through fontconfig. Your options are:

- Set fonts in the browser's own Settings → Appearance section (look for "Customize fonts" or similar).
- Accept that Chromium will use its own heuristics, which are generally reasonable but not configurable at the fontconfig level.
- For Electron apps specifically, some respect the `GTK_THEME` and system GTK font settings, so setting fonts via GNOME Settings or `~/.config/gtk-3.0/settings.ini` may have some effect on UI text — but not on web content rendering.

## 8. Understanding the conf.d Load Order in Depth

### How Files Are Sorted and Processed

When fontconfig scans a `conf.d` directory, every file whose name starts with an ASCII digit and ends with `.conf` is processed in sorted (lexicographic) order. In practice this means the two leading digits act as a priority prefix. `10-foo.conf` is processed before `20-bar.conf`, which is processed before `90-baz.conf`. The sort is purely string-based, so `09-x.conf` comes before `10-x.conf`, which comes before `99-x.conf`.

The fontconfig project has formalized these ranges into conventions:

| Prefix range | Purpose                                                                 |
| ------------ | ----------------------------------------------------------------------- |
| `00–09`      | Font directory declarations                                             |
| `10–19`      | System rendering defaults (antialiasing, hinting)                       |
| `20–29`      | Font rendering options                                                  |
| `30–39`      | Family substitution rules                                               |
| `40–49`      | Generic identification — mapping family names to generic types          |
| `50–59`      | Alternate config file loading (this is where user config is bridged in) |
| `60–69`      | Generic aliases — mapping generic names to specific families            |
| `70–79`      | Font availability selection                                             |
| `80–89`      | `target="scan"` rules — modifying scanned font patterns                 |
| `90–99`      | Font synthesis                                                          |

This means a file at `30-metric-aliases.conf` sets up family substitutions before `60-latin.conf` maps generic aliases, and so on. The ordering is not just a convention — it reflects the _logical pipeline_ fontconfig uses to resolve a font request from start to finish.

### The Critical Inversion: Lower Number ≠ Always Wins

Here is the most common source of confusion when writing conf.d snippets. Fontconfig will sort the ruleset files dropped in `/etc/fonts/conf.d/` by filename before evaluating them. Therefore the rules of `01-xxx.conf` and `02-yyy.conf` will have the same effect as a single file containing first the `xxx` rules then the `yyy` rules. Usually that means the files with the _smaller_ prefix will have higher precedence.

"Higher precedence" is true for `<alias>` blocks that use `<prefer>` — items prepended by `01-foo.conf` end up before items prepended by `60-bar.conf` in the final candidate list, because the first file ran first and pushed its entries to the front.

_But_ for `<match>` rules that use `mode="prepend"` on the `family` property, the logic inverts: since it uses the prepend mode, it follows an inverted priority — rulesets with high numeral prefixes take precedence over rulesets with low numeral prefixes. The reason is mechanical: both files call `prepend` on the same list. The file that ran _last_ (higher number) gets its entry pushed to the front most recently, so it wins. Think of it as a stack — each `prepend` pushes to the top, and the last push ends up on top.

This is why Fedora and other distros advise using high-numbered prefixes (e.g., `65–69`) for font-specific `<match>` rules that need to win over system defaults defined at `60–69`.

The practical rule of thumb:

- For `<alias>/<prefer>` blocks: **lower number wins** (runs first, entries are at the head of the list before later files add theirs).
- For `<match mode="prepend">` blocks: **higher number wins** (runs last, pushes its entries to the front of whatever earlier rules built up).

### The 50-user.conf Bridge: How Your Config Gets Loaded

There is a specific system-level snippet, `50-user.conf`, whose sole purpose is to load per-user customization files stored in XDG Base Directory spec-compliant locations — usually `$HOME/.config/fontconfig/conf.d` and `$HOME/.config/fontconfig/fonts.conf`.

The `50-user.conf` preset needs to be enabled to make user configuration take effect. User rules defined in `$XDG_CONFIG_HOME/fontconfig/fonts.conf` and in `$XDG_CONFIG_HOME/fontconfig/conf.d` are loaded via this file and typically take precedence over rules defined in files starting with a higher number.

You can verify this is working by running:

```bash
FC_DEBUG=1024 fc-match serif 2>&1 | grep -i "loading\|fontconfig"
```

You should see your `~/.config/fontconfig/fonts.conf` appear in the output. If it doesn't, check that `50-user.conf` is symlinked into `/etc/fonts/conf.d/` — on some minimal or embedded setups it may not be:

```bash
ls -la /etc/fonts/conf.d/ | grep 50-user
# Should show: 50-user.conf -> /usr/share/fontconfig/conf.avail/50-user.conf
```

If the symlink is missing, create it:

```bash
sudo ln -s /usr/share/fontconfig/conf.avail/50-user.conf /etc/fonts/conf.d/50-user.conf
sudo fc-cache -fv
```

Because `50-user.conf` sits in the `50–59` (alternate config loading) range, your personal files are processed _after_ all the system rendering defaults (10–49) and _before_ the final generic alias resolution (60–69). This gives your user-level rules a strong position in the pipeline — you can override rendering settings that were set at `10–19`, and you can influence alias resolution that happens at `60–69`.

### Checking What's Actually Enabled

On Debian and derivatives, many fontconfig snippets in `/usr/share/fontconfig/conf.avail/` are only active if symlinked into `/etc/fonts/conf.d/`. To see what's currently enabled:

```bash
ls /etc/fonts/conf.d/
```

Typical entries you'll see on a standard desktop:

```
10-hinting-slight.conf -> ../conf.avail/10-hinting-slight.conf
10-sub-pixel-rgb.conf  -> ../conf.avail/10-sub-pixel-rgb.conf
11-lcdfilter-default.conf -> ../conf.avail/11-lcdfilter-default.conf
20-unhint-small-vera.conf -> ../conf.avail/20-unhint-small-vera.conf
30-metric-aliases.conf -> ../conf.avail/30-metric-aliases.conf
45-generic.conf -> ../conf.avail/45-generic.conf
45-latin.conf   -> ../conf.avail/45-latin.conf
50-user.conf    -> ../conf.avail/50-user.conf
51-local.conf   -> ../conf.avail/51-local.conf
65-nonlatin.conf -> ../conf.avail/65-nonlatin.conf
69-unifont.conf  -> ../conf.avail/69-unifont.conf
```

On Arch Linux, `conf.d` files are managed differently — they're real files or symlinks under `/etc/fonts/conf.d/` and the `eselect fontconfig` tool (on Gentoo) or direct symlink management (on Arch) controls what's active.

### Naming Your Own Files

When adding your own snippets to `~/.config/fontconfig/conf.d/`, name them deliberately. A file like `99-my-overrides.conf` is guaranteed to run last among your user files, which is useful when you want your rules to take final precedence via `prepend`. A file like `01-rendering.conf` runs first and is appropriate for rendering defaults (hinting, anti-aliasing) that other rules should be able to see but not need to override.

If you want to prioritize between files with the same tens-prefix, you can add another number after the prefix — for example, `65-0-myfont.conf` will sort between `65-...` files in a predictable way.

## 9. Firefox Font Configuration: Beyond fontconfig

Firefox warrants special attention because it partially bypasses fontconfig in ways that surprise many users. Understanding exactly where Firefox listens and where it doesn't will save you hours of debugging.

### What Firefox Respects from fontconfig

Firefox _does_ use fontconfig for discovering which fonts are installed on the system. When it asks "is Inter available?", it queries fontconfig. However, Firefox runs its own internal font matching engine (based on Gecko's text layout) rather than delegating final selection back to fontconfig's substitution rules. This means your `<alias>` and `<match>` blocks have limited effect on how Firefox selects fonts for _web page content_.

Firefox has a setting which determines how many replacements it will allow from fontconfig. To allow it to use all your replacement rules, change `gfx.font_rendering.fontconfig.max_generic_substitutions` to `127` (the highest possible value). The default is much lower, which is why your fontconfig fallback chains may seem to be truncated inside Firefox. This is one of the first `about:config` tweaks to make.

### Setting Fonts via Preferences UI (the Right Starting Point)

Before reaching for `about:config`, try Firefox's built-in font preferences. Navigate to **Settings → General → Language and Appearance → Fonts**, then click **Advanced**. Here you can set:

- **Proportional**: the base font type (serif or sans-serif) and size
- **Serif**: your preferred serif font for web content
- **Sans-serif**: your preferred sans-serif font for web content
- **Monospace**: your preferred monospace font and size

You can also uncheck **"Allow pages to choose their own fonts, instead of your selections above"** to force your preferences onto all web pages — useful if you care more about readability consistency than respecting designer intent.

### Critical `about:config` Keys for Font Control

Open `about:config` (accept the warning), then search for these keys. Right-click any key and choose **Reset** to revert to default, or double-click a boolean to toggle it.

**`gfx.font_rendering.fontconfig.max_generic_substitutions`** (integer, default: `3`)

As noted above, this caps how many fontconfig substitutions Firefox will follow when resolving a generic family like `sans-serif`. Set it to `127` to allow the full chain defined in your fontconfig rules.

**`font.name-list.emoji`** (string)

Firefox ships with the Twemoji Mozilla font. To use the system emoji font instead, set `font.name-list.emoji` to `emoji` in `about:config`. Additionally, to prevent the bundled Mozilla font from interfering with your system emoji font, change `gfx.font_rendering.opentype_svg.enabled` to `false`, or remove `/usr/lib/firefox/fonts/TwemojiMozilla.ttf`.

This is critical if you've set up Apple Color Emoji or Noto Color Emoji via fontconfig — without this change, Firefox will use its bundled Twemoji for emoji regardless of your system configuration.

**`layout.css.dpi`** (integer, default: `0`)

Firefox by default uses 96 DPI and only uses the system's DPI if it is a higher value. To force the system's DPI regardless of its value, set `layout.css.dpi` to `0`. On HiDPI setups where your system DPI is less than 96 (unusual but possible in certain VM or display configurations), this matters.

**`layout.css.devPixelsPerPx`** (float, default: `-1.0`)

This controls the CSS pixel density for web content. The default of `-1.0` means "use the system setting." A solution for HiDPI displays is to change `layout.css.devPixelsPerPx` to the system DPI divided by 96. For example, if your system DPI is 144, set this to `1.5` to make web content use a DPI of 144.

**`gfx.font_rendering.opentype_svg.enabled`** (boolean, default: `true`)

When `true`, Firefox uses its internal OpenType SVG renderer, which is what makes its bundled TwemojiMozilla font render as color emoji. Setting this to `false` disables OpenType SVG rendering entirely. The tradeoff: you lose Firefox's bundled Twemoji, but fontconfig-configured emoji then work correctly. If you want system emoji without removing the Firefox font file, this toggle is your cleanest option.

Here's a summary table for quick reference:

| `about:config` key                                        | Recommended value | Why                                               |
| --------------------------------------------------------- | ----------------- | ------------------------------------------------- |
| `gfx.font_rendering.fontconfig.max_generic_substitutions` | `127`             | Honor full fontconfig substitution chains         |
| `font.name-list.emoji`                                    | `emoji`           | Use system emoji font instead of bundled Twemoji  |
| `gfx.font_rendering.opentype_svg.enabled`                 | `false`           | Prevent bundled font from overriding system emoji |
| `layout.css.dpi`                                          | `0`               | Respect system DPI exactly                        |

### Controlling Firefox's UI Fonts

Firefox's UI (tab bar, address bar, menus) is rendered separately from web content. On Linux, the `widget.use-xdg-desktop-portal.font-settings` setting and GTK font preferences affect the user interface, but do not affect webpages — only the user interface itself. The UI font follows the GTK font setting, which you can change via:

```bash
# Set GTK3 interface font to Inter 11pt
gsettings set org.gnome.desktop.interface font-name "Inter 11"
```

For finer UI control, Firefox supports a `userChrome.css` file that can override specific UI elements with CSS font rules. To enable it, first set `toolkit.legacyUserProfileCustomizations.stylesheets` to `true` in `about:config`, then create:

```
~/.mozilla/firefox/<your-profile>/chrome/userChrome.css
```

A minimal example that forces your UI font:

```css
/* Force Inter for all Firefox UI text */
* {
  font-family: 'Inter', sans-serif !important;
}

/* Specifically target the address bar */
#urlbar-input {
  font-family: 'JetBrains Mono', monospace !important;
}
```

This gives you pixel-level control over Firefox's interface typography, independent of both fontconfig and GTK settings.

### Firefox and Chromium: The Core Difference

To be direct: **Firefox is the only major browser on Linux that gives you meaningful fontconfig integration** for web content. Chromium simply ignores fontconfig for the most part. Chrome, Brave, Edge, and Vivaldi use Skia's own font fallback logic and query the system font list but do not honor your substitution or alias rules. For font control in Chromium-based browsers, use the browser's built-in **Settings → Appearance → Customize fonts** and accept that fontconfig rules won't propagate into them.

## 10. Troubleshooting: Emoji Rendering as Black & White in Firefox

This is one of the most commonly reported and most confusing font problems on Linux. You have a color emoji font installed, `fc-match emoji` returns the right font, emoji look fine in your terminal and GTK apps — but open Firefox and some emoji render as flat, monochrome outlines. The classic example is the 🌶️ hot pepper emoji, which shows up as a black-and-white sketch rather than the expected bright red chili.

There is not one cause for this. There are at least four distinct root causes, each with a different fix. Working through them in order is the fastest path to resolution.

---

### Root Cause 0: The Hot Pepper Is _Supposed_ to Be Monochrome (The Unicode Variation Selector Problem)

Before debugging fontconfig at all, check whether the emoji in question is a _text-presentation_ codepoint. The Unicode standard allows certain emoji to have both a text (monochrome) and emoji (color) presentation. The hot pepper 🌶 (U+1F336) is one such character — it will be rendered as a monochrome character on conforming platforms unless it is followed by **Variation Selector-16** (U+FE0F), which explicitly requests the emoji presentation. The sequence U+1F336 + U+FE0F — rendered as 🌶️ — is what tells a browser and font renderer to use the color glyph.

Many websites, chat applications, and copy-paste sources omit the variation selector, meaning the character in the HTML source is literally just `🌶` with no selector. A standards-compliant renderer then correctly serves the monochrome glyph. Firefox is particularly strict about this.

**How to check:** Open Firefox's developer tools (F12), select the element containing the emoji, and look at the raw character in the HTML source. If it's bare U+1F336 without U+FE0F appended, the monochrome rendering is not a bug — it's correct behavior. The fix belongs with whoever controls the content, not your font configuration.

However, many users still prefer color rendering even for text-presentation emoji. Some emoji fonts (notably Noto Color Emoji from version 2.038 onwards) include special fontconfig metadata to prefer color presentation, and some browser-level workarounds exist. If you want color everywhere regardless of variation selectors, the fontconfig charset approach in Section 5 combined with the `about:config` tweaks in Section 9 is your best lever.

---

### Root Cause 1: `embeddedbitmap false` Is Silently Suppressing Color Emoji in Firefox

This is the most widespread cause and the one that catches the most people, because the fix they applied for one problem (disabling ugly legacy bitmap fonts) breaks something else entirely.

The standard advice for disabling old bitmap fonts is to add `embeddedbitmap false` globally to your fontconfig:

```xml
<!-- This is the PROBLEMATIC config that breaks emoji in Firefox -->
<match target="font">
  <edit name="embeddedbitmap" mode="assign">
    <bool>false</bool>
  </edit>
</match>
```

This looks innocent and works fine in every GTK/Qt application. But Firefox uses a patched version of Cairo for font rendering, and Cairo interprets the `embeddedbitmap = false` flag as an instruction to not display the embedded bitmaps in a selected font — meaning a color emoji font whose glyphs _are_ bitmaps gets told by Firefox to render nothing at all.

The result: some emoji are missing, show as boxes, or fall back to a monochrome outline font that happens to have those codepoints.

**The fix** is to carve out a specific exception for your color emoji font so it re-enables embedded bitmaps after the global rule disables them:

```xml
<!-- Global rule: disable embedded bitmaps for all fonts (improves text rendering) -->
<match target="font">
  <edit name="embeddedbitmap" mode="assign">
    <bool>false</bool>
  </edit>
</match>

<!-- Exception: re-enable embedded bitmaps for color emoji fonts.
     Without this, Firefox/Cairo suppresses color emoji rendering entirely. -->
<match target="font">
  <test qual="any" name="family">
    <string>Noto Color Emoji</string>
  </test>
  <edit name="embeddedbitmap" mode="assign">
    <bool>true</bool>
  </edit>
</match>

<!-- Add a second block if using Apple Color Emoji -->
<match target="font">
  <test qual="any" name="family">
    <string>Apple Color Emoji</string>
  </test>
  <edit name="embeddedbitmap" mode="assign">
    <bool>true</bool>
  </edit>
</match>
```

The order matters here: the global `false` rule must come first, and the per-family `true` exception must come after it in document order (or in a higher-numbered conf.d file) so it overwrites the global setting for the emoji font specifically.

After making this change, rebuild the cache and restart Firefox completely (not just a new window — quit and relaunch):

```bash
fc-cache -fv
# Then fully quit and relaunch Firefox
```

---

### Root Cause 2: A Competing Font Is Stealing Emoji Codepoints

Some widely-installed fonts contain a subset of emoji codepoints as monochrome outlines. The classic offenders are **DejaVu Sans**, **Symbola**, and certain versions of **GNU FreeFont**. DejaVu fonts and Symbola conflict with your dedicated emoji font and cause some emoji to be displayed as monochrome.

This happens because fontconfig's fallback chain reaches your text font _before_ your emoji font for certain codepoints — specifically the ones your text font happens to include. The text font wins, serves a monochrome glyph, and your color emoji font never gets called.

**Diagnosing it:** Use Firefox's built-in font inspector. Open the developer tools (F12), go to the **Inspector** tab, right-click on any emoji element and choose **Inspect**, then in the **Fonts** tab (or the **Computed** tab → scroll to `font-family`) you'll see which font Firefox actually used to render that character. If it shows "DejaVu Sans" or "Symbola" for an emoji, you've found the culprit.

You can also check via the command line. The `fc-match` approach from Section 4 is useful:

```bash
# Check what font would be used for the hot pepper codepoint U+1F336
python3 -c "print('\U0001F336')" | fc-match --sort -
```

**The fix** is the charset-stripping approach from Section 5, but you need to be explicit about which fonts to exclude. The safest version lists specific offending families:

```xml
<!-- Strip emoji codepoints from known conflicting text fonts -->
<match target="font">
  <test name="family" compare="eq">
    <string>DejaVu Sans</string>
  </test>
  <edit name="charset" mode="assign">
    <minus>
      <n>charset</n>
      <charset>
        <range><int>0x1F300</int><int>0x1F9FF</int></range>
        <range><int>0x2600</int><int>0x27BF</int></range>
      </charset>
    </minus>
  </edit>
</match>

<match target="font">
  <test name="family" compare="eq">
    <string>Symbola</string>
  </test>
  <edit name="charset" mode="assign">
    <minus>
      <n>charset</n>
      <charset>
        <range><int>0x1F300</int><int>0x1F9FF</int></range>
        <range><int>0x2600</int><int>0x27BF</int></range>
        <range><int>0x1F000</int><int>0x1F02F</int></range>
      </charset>
    </minus>
  </edit>
</match>
```

If you rarely use Symbola for its intended purpose (covering obscure Unicode blocks), you can alternatively uninstall it entirely:

```bash
# Debian/Ubuntu
sudo apt remove fonts-symbola

# Arch
sudo pacman -R ttf-symbola
```

---

### Root Cause 3: Multiple Emoji Fonts Are Installed and Conflicting

If you have more than one emoji font installed, they can conflict and cause inconsistent rendering — some emoji color, some monochrome. This is particularly common on Arch where users install multiple packages from the AUR, or on Ubuntu where `fonts-noto-color-emoji` and `ttf-twemoji` may both be present.

**Diagnosing it:**

```bash
# List all installed emoji fonts
fc-list | grep -i emoji

# Typical messy output on a system with conflicts:
# /usr/share/fonts/noto/NotoColorEmoji.ttf: Noto Color Emoji:style=Regular
# /usr/share/fonts/TTF/TwemojiMozilla.ttf: Twemoji Mozilla:style=Regular
# /usr/share/fonts/truetype/ancient-scripts/Symbola_hint.ttf: Symbola:style=Regular
```

**The fix** depends on which emoji font you want to keep. The cleanest resolution is to uninstall the ones you don't want:

```bash
# Keep only Noto Color Emoji on Debian/Ubuntu
sudo apt install fonts-noto-color-emoji
sudo apt remove fonts-twemoji  # if installed

# Rebuild cache after cleanup
fc-cache -fv
```

If you can't or don't want to uninstall, you can use fontconfig substitution to redirect all requests for competing emoji fonts to your preferred one:

```xml
<!-- Redirect all emoji font variants to your preferred choice -->
<match target="pattern">
  <test qual="any" name="family">
    <string>Twemoji Mozilla</string>
  </test>
  <edit name="family" mode="assign" binding="same">
    <string>Noto Color Emoji</string>
  </edit>
</match>

<match target="pattern">
  <test qual="any" name="family">
    <string>Twemoji</string>
  </test>
  <edit name="family" mode="assign" binding="same">
    <string>Noto Color Emoji</string>
  </edit>
</match>
```

---

### Root Cause 4: Firefox Is Using Its Bundled TwemojiMozilla Font Instead of the System Font

Firefox ships with a copy of `TwemojiMozilla.ttf` inside its installation directory (typically `/usr/lib/firefox/fonts/` on Linux). This font is loaded at a very high priority and will override your system emoji configuration for any codepoints it covers. If your system emoji font and the bundled Twemoji don't have consistent coverage, you'll see a mix: some emoji render in Twemoji style, others fall through to your system font.

**Diagnosing it:** In Firefox's developer tools → Inspector → Fonts tab, if you see "Twemoji Mozilla" listed for emoji you expected to render as Noto or Apple Color Emoji, this is the cause.

**Fix option 1 (preferred): Disable OpenType SVG rendering in Firefox.**

As covered in Section 9, setting `gfx.font_rendering.opentype_svg.enabled` to `false` in `about:config` disables the SVG-layer color rendering that Firefox's bundled Twemoji uses. Firefox will then fall back to fontconfig for emoji, which respects your system configuration.

**Fix option 2: Remove the bundled font file.**

If you need OpenType SVG rendering for other purposes (rare on Linux), you can delete or rename the bundled font directly:

```bash
# Debian/Ubuntu Firefox from apt
sudo mv /usr/lib/firefox/fonts/TwemojiMozilla.ttf \
        /usr/lib/firefox/fonts/TwemojiMozilla.ttf.bak

# Arch Linux Firefox from pacman
sudo mv /usr/lib/firefox/fonts/TwemojiMozilla.ttf \
        /usr/lib/firefox/fonts/TwemojiMozilla.ttf.bak
```

Note: package updates will restore this file. Treat it as a temporary fix or script it as a post-install hook.

**Fix option 3: Explicitly override the `font.name-list.emoji` setting.**

In `about:config`, set `font.name-list.emoji` to the exact family name of your preferred emoji font (e.g., `Noto Color Emoji` or `Apple Color Emoji`). This tells Firefox's Gecko layout engine to prefer your named font for the emoji generic family, above the bundled Twemoji.

---

### Quick Diagnostic Checklist

When emoji render in black and white in Firefox, work through these steps in order:

```bash
# Step 1: Check if the emoji uses a variation selector
# Paste the emoji into this Python snippet — does U+FE0F appear?
python3 -c "s='🌶️'; print([hex(ord(c)) for c in s])"
# Color: ['0x1f336', '0xfe0f']  — has selector, should be color
# Mono:  ['0x1f336']            — no selector, monochrome is correct

# Step 2: Check which font fontconfig selects for emoji
fc-match emoji
# Should return your color emoji font, not DejaVu or Symbola

# Step 3: Check for conflicting emoji fonts
fc-list | grep -i "emoji\|symbola\|noto.*emoji\|twemoji"

# Step 4: Verify embeddedbitmap is re-enabled for your emoji font
fc-match -v "Noto Color Emoji" | grep embeddedbitmap
# Should show: embeddedbitmap: True(w)

# Step 5: Open Firefox developer tools on a page with the broken emoji
# Inspector → select the emoji element → Fonts tab
# Note which font Firefox reports using
```

If Step 5 shows Firefox using the right font but still rendering monochrome, the embeddedbitmap issue (Root Cause 1) is almost certainly responsible. If it shows the wrong font entirely, you have a codepoint-stealing problem (Root Cause 2 or 3). If it shows "Twemoji Mozilla", go to Root Cause 4.

---

### The Complete Fix: All Guards Together

For a system where you want robust, consistent color emoji in Firefox with the fewest surprises, this fontconfig snippet combines all the fixes in one place. Add it as `~/.config/fontconfig/conf.d/98-emoji-firefox-fix.conf`:

```xml
<?xml version="1.0"?>
<!DOCTYPE fontconfig SYSTEM "fonts.dtd">
<fontconfig>

  <!-- 1. Global: disable embedded bitmaps in text fonts (standard quality fix) -->
  <match target="font">
    <edit name="embeddedbitmap" mode="assign">
      <bool>false</bool>
    </edit>
  </match>

  <!-- 2. Exception: re-enable embedded bitmaps for color emoji fonts.
          This is the fix for Firefox/Cairo suppressing color emoji. -->
  <match target="font">
    <test qual="any" name="family">
      <string>Noto Color Emoji</string>
    </test>
    <edit name="embeddedbitmap" mode="assign"><bool>true</bool></edit>
    <!-- Also disable hinting and antialias — they don't apply to bitmap glyphs
         and having them enabled can confuse Firefox's Cairo rendering layer -->
    <edit name="hinting" mode="assign"><bool>false</bool></edit>
    <edit name="antialias" mode="assign"><bool>false</bool></edit>
  </match>

  <!-- Repeat for Apple Color Emoji if you use it instead -->
  <match target="font">
    <test qual="any" name="family">
      <string>Apple Color Emoji</string>
    </test>
    <edit name="embeddedbitmap" mode="assign"><bool>true</bool></edit>
    <edit name="hinting" mode="assign"><bool>false</bool></edit>
    <edit name="antialias" mode="assign"><bool>false</bool></edit>
  </match>

  <!-- 3. Strip emoji codepoints from DejaVu Sans (common codepoint thief) -->
  <match target="font">
    <test name="family" compare="eq">
      <string>DejaVu Sans</string>
    </test>
    <edit name="charset" mode="assign">
      <minus>
        <n>charset</n>
        <charset>
          <range><int>0x1F300</int><int>0x1F9FF</int></range>
          <range><int>0x2600</int><int>0x26FF</int></range>
          <range><int>0x2700</int><int>0x27BF</int></range>
        </charset>
      </minus>
    </edit>
  </match>

  <!-- 4. Strip emoji codepoints from Symbola -->
  <match target="font">
    <test name="family" compare="eq">
      <string>Symbola</string>
    </test>
    <edit name="charset" mode="assign">
      <minus>
        <n>charset</n>
        <charset>
          <range><int>0x1F000</int><int>0x1FFFF</int></range>
          <range><int>0x2600</int><int>0x27BF</int></range>
        </charset>
      </minus>
    </edit>
  </match>

  <!-- 5. Redirect competing emoji font names to your preferred font -->
  <match target="pattern">
    <test qual="any" name="family">
      <string>Twemoji Mozilla</string>
    </test>
    <edit name="family" mode="assign" binding="same">
      <string>Noto Color Emoji</string>
    </edit>
  </match>

</fontconfig>
```

After saving, rebuild the cache and fully restart Firefox:

```bash
fc-cache -fv
# Quit Firefox entirely, then relaunch
```

Then set these two keys in `about:config`:

- `gfx.font_rendering.opentype_svg.enabled` → `false`
- `gfx.font_rendering.fontconfig.max_generic_substitutions` → `127`

Between the fontconfig fix and the two `about:config` keys, nearly every instance of the black-and-white emoji problem in Firefox can be resolved.
